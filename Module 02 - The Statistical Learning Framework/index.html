  • Define mean squared error and explain why training error alone is a misleading measure of a model

Recall, Why We Estimate f

In Module 1 we said machine learning is concerned with algorithms that improve their performance on a task using data. In the statistical learning framework, that idea is sharpened into one specific job: estimating an unknown function from observed data.

We assume there is some true, unknown relationship between an input and an output, and our job is to recover a useful approximation of it using a finite sample of observations.

The two reasons to estimate f, from Module 1: prediction, where we mainly want an accurate output for new inputs, and inference, where we mainly want to understand the relationship itself.

Everything in this module is about doing that estimation carefully, and about judging honestly how good the result is.

The Framework, Formally

We write the relationship between an input \(X\) and an output \(Y\) as

\[ Y = f(X) + \varepsilon \]

Here is what each symbol means, precisely.

SymbolMeaning
\(X\)The input variable, or vector of input variables, also called features or predictors
\(Y\)The output, or target, or response, the quantity we want to predict or explain
\(f\)The true, fixed but unknown function relating \(X\) to \(Y\)
\(\varepsilon\)A random error term, independent of \(X\), with mean zero

Statistical learning is a collection of approaches for estimating \(f\), which we write as \(\hat{f}\), using observed pairs \((x_1, y_1), \dots, (x_N, y_N)\).

Where Epsilon Comes From

The error term \(\varepsilon\) is not a mistake in our model, it is a permanent feature of almost every real prediction problem. It exists for reasons such as

  • Unmeasured variables: factors that affect \(Y\) but were never recorded, for instance a customer's mood on the day they decided to cancel a mobile data plan
  • Measurement error: the recorded value of \(X\) or \(Y\) is not perfectly accurate, for instance a rain gauge reading that is slightly off
  • Genuine randomness: some outcomes may simply not be fully determined by anything we could measure in advance
Important consequence: because \(\varepsilon\) exists, even if we somehow learned the true \(f\) exactly, our predictions would still not be perfect. This sets a floor on how good any model can possibly be, called the irreducible error, which we return to later in this module.

Measuring Model Accuracy, Mean Squared Error

Once we have an estimate \(\hat{f}\), we need a number that tells us how good it is. In a regression setting, the standard measure is the mean squared error, MSE.

\[ MSE = \frac{1}{n} \sum_{i=1}^{n} \left( y_i - \hat{f}(x_i) \right)^2 \]

MSE averages the squared difference between each true value \(y_i\) and the model's prediction \(\hat{f}(x_i)\). Squaring serves two purposes, it makes every term positive, so overestimates and underestimates do not cancel out, and it penalizes large errors more heavily than small ones.

Small MSE is good: it means predictions are, on average, close to the actual values, in the units of \(Y\) squared.

A Worked MSE Calculation

Suppose a model predicts the number of daily transactions (in hundreds) at five mobile banking agent points, and we compare predictions to what actually happened.

The sum of squared errors is \(1+1+1+1+1 = 5\), and \(n = 5\), so

\[ MSE = \frac{5}{5} = 1 \]

On average, the model's predictions are off by an amount whose square is 1, here exactly one hundred transactions squared, since each unit represents a hundred transactions.

Training MSE Versus Test MSE

The MSE computed on the same data used to fit the model is called the training MSE. It is tempting to just pick whichever model has the smallest training MSE. This is a trap.

The core problem: there is no guarantee that the method with the smallest training MSE will also have the smallest test MSE, the error measured on fresh data the model has never seen.

A model can achieve a very low training MSE simply by becoming flexible enough to memorize the noise in the training data, rather than learning the true underlying pattern. What we actually care about is performance on new data, since that is what the model will face in practice, a new customer, a new week of rainfall, a new transaction.

The U-Shaped Test Error Curve

If we plot error against model flexibility, training MSE and test MSE behave very differently.

Training MSE

  • Decreases steadily as flexibility increases
  • A sufficiently flexible model can drive training MSE arbitrarily close to zero
  • By itself, never tells you to stop adding flexibility

Test MSE

  • Decreases at first, as flexibility helps capture real structure
  • Reaches a minimum at some intermediate flexibility
  • Increases again as the model starts fitting noise, this is overfitting

This is the single most important picture in this module. The shape is U-shaped for test error, and monotonically decreasing for training error, and the gap between the two curves is itself meaningful, a wide gap signals overfitting.

Bias and Variance, Revisited and Formalized

Module 1 introduced bias and variance conceptually. Now we define them precisely, as properties of the estimator \(\hat{f}\) itself, not of any single prediction.

TermFormal meaning
BiasThe error introduced by approximating a possibly complex true \(f\) with a simpler model
VarianceThe amount by which \(\hat{f}\) would change if estimated using a different training set

Both are properties of the fitting procedure, not the data. If you retrained the same procedure on a different sample of customers or transactions, a high variance procedure would produce a noticeably different \(\hat{f}\), while a high bias procedure would produce a similarly inaccurate one each time.

The Bias Variance Decomposition

For a given test point \(x_0\), the expected test MSE can be written exactly as the sum of three non negative quantities.

\[ E\left[\left(y_0 - \hat{f}(x_0)\right)^2\right] = \text{Var}\left(\hat{f}(x_0)\right) + \left[\text{Bias}\left(\hat{f}(x_0)\right)\right]^2 + \text{Var}(\varepsilon) \]

The expectation on the left is taken over many different training sets, it asks, on average across all the training sets we could have drawn, how far off is our prediction at \(x_0\)?

Why this is useful: it tells us that to get low expected test error, a method needs both low variance and low bias at the same time, and that no method can ever push the right hand side below \(\text{Var}(\varepsilon)\).

Reading the Decomposition, Term by Term

  • \(\text{Var}(\hat{f}(x_0))\), variance: how much the prediction at \(x_0\) would bounce around if we refit the model on many different training sets drawn from the same population
  • \([\text{Bias}(\hat{f}(x_0))]^2\), squared bias: how far the average prediction, averaged over those same many training sets, is from the truth \(f(x_0)\)
  • \(\text{Var}(\varepsilon)\), irreducible error: the variance of the noise itself, present no matter how good the model is
A subtlety worth noting: bias is squared in this formula, so it is always non negative, exactly like variance, which means all three terms add up, none of them can partially cancel another out.

Irreducible Error, the Floor We Cannot Go Below

\(\text{Var}(\varepsilon)\) is called the irreducible error because it comes from the randomness in \(Y = f(X) + \varepsilon\) itself, not from any weakness in our chosen model or fitting procedure.

No method can beat it: even an estimator that recovered the true \(f\) exactly, with zero bias and zero variance, would still have an expected squared error of exactly \(\text{Var}(\varepsilon)\) at every point.

In practice we never know \(\text{Var}(\varepsilon)\) exactly, since we never observe \(f\) directly. But recognizing that it exists changes how we interpret a model's error, an error of a certain size might already be close to the best any model could ever achieve on that problem.

How Flexibility Drives Bias and Variance

Model flexibility connects directly to the two reducible terms in the decomposition.

Less flexible models

  • Make strong assumptions about the shape of \(f\)
  • Tend to have higher bias, if the true \(f\) is more complex than assumed
  • Tend to have lower variance, since they do not chase every detail of the training sample

More flexible models

  • Make weaker assumptions, can approximate complex shapes
  • Tend to have lower bias, when the true relationship is genuinely complex
  • Tend to have higher variance, since they are sensitive to the specific training sample

This is exactly why the test MSE curve is U-shaped, as flexibility increases from very low, squared bias falls quickly at first while variance rises slowly, then the trend reverses, variance starts dominating and total error climbs back up.

Worked Example, Bias and Variance for a Flood Risk Model

Recall the flood early warning model from Module 1. Suppose we compare two fitting procedures.

  • Model A: predicts the same average flood probability for every district, ignoring rainfall and river level entirely
  • Model B: a highly flexible model that fits this season's rainfall and river level data almost perfectly, district by district

Model A has high bias, it is too simple to capture how rainfall actually drives flood risk, so it is systematically wrong in the same direction every season. Model B has high variance, fit it on a different season's data and the predicted risk pattern would look quite different, even though both seasons follow the same true underlying relationship.

What we actually want: a model with enough flexibility to use rainfall and river level meaningfully, district by district, but not so much flexibility that it starts reacting to one season's particular weather quirks as if they were the rule.

A Concrete Illustration, Polynomial Curve Fitting

The bias variance tradeoff can feel abstract until you see it play out in a single, fully worked example. We will use a classic one, fitting a polynomial curve to noisy data generated from a known underlying pattern.

Imagine we generate data points from a smooth wave like pattern, then add a small amount of random noise to each point, mimicking measurement error. We then try to recover the underlying pattern using polynomials of different orders \(M\).

Why a synthetic example helps: because we generated the data ourselves, we actually know the true underlying function. That lets us see exactly when a fitted model is capturing the truth, and when it is instead fitting noise, something we almost never get to see this clearly with real data.

Specifying the Polynomial Model

A polynomial of order \(M\) takes the form

\[ y(x, w) = w_0 + w_1 x + w_2 x^2 + \cdots + w_M x^M = \sum_{m=0}^{M} w_m x^m \]

The order \(M\) controls flexibility directly.

  • \(M = 0\) is a flat horizontal line, the least flexible choice, just one parameter, \(w_0\)
  • \(M = 1\) is a straight line, slightly more flexible, two parameters
  • Larger \(M\) allows more bends and wiggles, with \(M+1\) parameters \(w_0, w_1, \dots, w_M\) to choose

The values \(w_0, \dots, w_M\) are called the coefficients, and fitting the model means choosing values for them.

The Error Function for Curve Fitting

We choose the coefficients \(w\) to minimize a sum of squared errors over the training points,

\[ E(w) = \frac{1}{2} \sum_{n=1}^{N} \left( y(x_n, w) - t_n \right)^2 \]

where \(t_n\) is the observed, noisy target value at \(x_n\), and \(y(x_n, w)\) is the polynomial's prediction at \(x_n\). This is exactly the same idea as MSE from earlier in this module, the factor of one half is just a convenience that simplifies the calculus used to minimize it, it does not change which \(w\) is best.

Connection: minimizing \(E(w)\) over the training points is the curve fitting version of minimizing training MSE, the same trap from slide 8 applies here too.

Comparing Model Order M, Underfitting to Overfitting

Order MBehaviorDiagnosis
0A flat line, completely misses the wave shaped patternSevere underfitting, high bias
1A straight line, still misses the curvature of the patternUnderfitting, high bias
3Follows the general wave shape closely, small training errorA good fit, reasonable bias and variance
9Passes through nearly every training point exactly, but oscillates wildly between themSevere overfitting, low bias, very high variance
The M = 9 case is instructive: training error at \(M = 9\) can be driven to almost zero, yet the fitted curve looks nothing like the true underlying pattern between the training points. The coefficients themselves often become extremely large, a telltale numerical sign of overfitting.

Root Mean Square Error

To compare fits across different sample sizes \(N\) on a common scale, it is common to report the root mean square error, RMS error, rather than the raw sum of squared errors.

\[ E_{RMS} = \sqrt{\frac{2 E(w^{*})}{N}} \]

Here \(w^{*}\) denotes the best fitting coefficients found by minimizing \(E(w)\), and \(N\) is the number of data points. Dividing by \(N\) makes the measure comparable across different sample sizes, and the square root puts the error back into the same units as the original target \(t\), which makes it easier to interpret directly.

Pattern to expect: training RMS error keeps falling as \(M\) increases. Test RMS error falls, then rises again once \(M\) grows large enough to overfit, exactly the U-shape from slide 9, now in the specific language of curve fitting.

More Data, Less Overfitting Risk

Overfitting at a given order \(M\) is not a fixed property of that order alone, it also depends heavily on how much training data \(N\) is available.

  • With a small \(N\), say 15 points, a high order polynomial such as \(M = 9\) has more than enough freedom to chase every point, severe overfitting results
  • With a much larger \(N\), say 100 points, the same \(M = 9\) polynomial is far more constrained, since it must now pass reasonably close to many more points at once, and the fitted curve looks much closer to the true underlying pattern
General principle: for a fixed model flexibility, more training data tends to reduce variance, since the fitting procedure has less freedom to latch onto the noise of any one small sample.

This is one reason large datasets have become so valuable, they allow more flexible models to be used safely.

A Preview of Regularization

When more data is not available, there is another way to control overfitting directly, by penalizing large coefficients in the error function itself.

\[ \tilde{E}(w) = \frac{1}{2} \sum_{n=1}^{N} \left( y(x_n, w) - t_n \right)^2 + \frac{\lambda}{2} \lVert w \rVert^2 \]

The added term \(\frac{\lambda}{2}\lVert w \rVert^2\) grows larger as the coefficients grow larger, so minimizing \(\tilde{E}(w)\) now means balancing two goals at once, fitting the training points well, and keeping the coefficients from becoming extreme. The constant \(\lambda \ge 0\) controls how strongly that second goal is enforced.

Looking ahead: this exact idea, penalizing coefficient size, reappears formally as ridge regression and the elastic net later in the course. For now, just recognize the shape of the idea, add a penalty, control flexibility without throwing away data.

From Regression to Classification, the Error Rate

Everything so far assumed a quantitative target. When the target \(Y\) is qualitative, a category rather than a number, MSE no longer makes sense, so we need an analogous measure.

\[ \text{Error rate} = \frac{1}{n} \sum_{i=1}^{n} I(y_i \ne \hat{y}_i) \]

Here \(I(y_i \ne \hat{y}_i)\) is an indicator that equals 1 when the predicted class \(\hat{y}_i\) does not match the true class \(y_i\), and 0 when it does. The error rate is simply the fraction of observations misclassified, and exactly like MSE, we care most about the test error rate, not the training error rate.

The Bayes Classifier and Bayes Error Rate

In classification, the theoretical best possible classifier assigns each observation to the class that is most probable given its predictor values. This is called the Bayes classifier, and the error rate it achieves is called the Bayes error rate.

The Bayes error rate is a floor, just like the irreducible error in regression: no classifier, however sophisticated, can do better than the Bayes error rate on average, because some overlap between classes is simply part of the true population.

In practice we never know the true class probabilities, so we cannot compute the Bayes classifier exactly, but it remains a useful theoretical benchmark, the best any method could possibly achieve on a given problem.

K-Nearest Neighbors, Flexibility Through K

To make the flexibility idea concrete in classification, consider K-Nearest Neighbors, KNN, a simple method that classifies a new point by looking at the \(K\) training points closest to it, then assigning the majority class among those \(K\) neighbors.

The choice of \(K\) controls flexibility directly, in the opposite direction from what you might first guess.

Choice of KFlexibilityTypical behavior
Small K, such as K = 1Very highDecision boundary is jagged, fits training data almost perfectly, high variance
Large K, close to NVery lowDecision boundary is smooth, close to a single overall majority vote, high bias
Why this matters here: KNN is not the focus of this module, it is simply the clearest way to see flexibility, bias, and variance acting in a classification setting before we study classification methods properly in Module 4.

Training Versus Test Error Rate as K Varies

Exactly as with polynomial order \(M\) and MSE, plotting error rate against \(1/K\), which increases as \(K\) decreases, reveals the same two familiar shapes.

Training error rate

  • Falls steadily as K decreases, more flexibility
  • Reaches zero at K = 1, every point is its own nearest neighbor

Test error rate

  • Falls at first as K decreases from a large value
  • Reaches a minimum at some intermediate K
  • Rises again as K becomes too small, the classifier overfits to individual training points

The lesson is the same lesson as the entire module so far, training performance alone never tells you when to stop adding flexibility.

Why We Need Probability Now

Notice how often the word probability has already appeared in this module, the random error \(\varepsilon\), the Bayes classifier choosing the most probable class, the irreducible error as a kind of fundamental uncertainty. To work with any of these ideas precisely, we need a small set of formal tools from probability theory.

The rest of this module builds exactly the tools used later, sum rule and product rule, Bayes' theorem, expectation, variance, and covariance, the same variance that appears inside the bias variance decomposition you just learned.

Running example: a mobile financial service flags some transactions as possibly fraudulent. We will use this single example throughout the probability section, since the same numbers can illustrate every rule in turn.

The Sum Rule and Product Rule

Suppose out of 10,000 transactions at a mobile financial service, we record whether each transaction was actually fraudulent, \(Y\), and whether the system flagged it, \(X\).

Flagged (X = 1)Not flagged (X = 0)Total
Fraud (Y = 1)15050200
Not fraud (Y = 0)15096509800
Total300970010000

From the counts, \(p(X{=}1, Y{=}1) = 150/10000 = 0.015\). The sum rule says we can recover the probability of one variable alone by summing the joint probability over all values of the other,

\[ p(X) = \sum_{Y} p(X, Y) \]

Checking it: \(p(X{=}1) = p(X{=}1,Y{=}1) + p(X{=}1,Y{=}0) = 0.015 + 0.015 = 0.03\), matching \(300/10000\) directly from the table.

The product rule relates the joint probability to a conditional probability,

\[ p(X, Y) = p(X \mid Y)\, p(Y) \]

Conditional Probability, a Worked Example

The conditional probability \(p(X \mid Y)\) means, restrict attention only to the cases where \(Y\) takes a particular value, then ask about \(X\) within that restricted group.

Using the same fraud table, what fraction of actual fraud cases get flagged?

\[ p(\text{Flagged} \mid \text{Fraud}) = \frac{p(\text{Flagged}, \text{Fraud})}{p(\text{Fraud})} = \frac{150/10000}{200/10000} = \frac{150}{200} = 0.75 \]

So 75 percent of true fraud cases are caught by the flag. Now ask the reverse question, of the cases the system flags, what fraction are actually fraud?

\[ p(\text{Fraud} \mid \text{Flagged}) = \frac{p(\text{Fraud}, \text{Flagged})}{p(\text{Flagged})} = \frac{150/10000}{300/10000} = \frac{150}{300} = 0.5 \]
These are different numbers for a reason: \(p(\text{Flagged} \mid \text{Fraud})\) and \(p(\text{Fraud} \mid \text{Flagged})\) answer different questions, confusing the two is one of the most common reasoning mistakes in applied statistics.

Bayes' Theorem

Bayes' theorem follows directly from the product rule, applied twice and rearranged, and lets us flip a conditional probability around,

\[ p(Y \mid X) = \frac{p(X \mid Y)\, p(Y)}{p(X)} \]

Let's verify it using the two conditional probabilities from the previous slide. We know \(p(\text{Flagged}\mid\text{Fraud}) = 0.75\), \(p(\text{Fraud}) = 0.02\), and \(p(\text{Flagged}) = 0.03\). Bayes' theorem says

\[ p(\text{Fraud} \mid \text{Flagged}) = \frac{0.75 \times 0.02}{0.03} = \frac{0.015}{0.03} = 0.5 \]

This matches the value we computed directly from the table on the previous slide, exactly as it must.

Why Bayes' theorem matters for this course: it is the formal basis for the Bayes classifier from slide 27, and it reappears explicitly in later modules whenever a method needs to convert a model of \(p(X \mid Y)\) into the more useful \(p(Y \mid X)\).

Expectation

The expectation of a function \(f(x)\) of a random variable, written \(E[f]\), is its average value, weighted by how probable each outcome is.

CaseFormula
Discrete \(x\)\( E[f] = \sum_x p(x) f(x) \)
Continuous \(x\)\( E[f] = \int p(x) f(x)\, dx \)

In practice, we rarely know \(p(x)\) exactly, so we approximate the expectation using a finite sample of \(N\) observed points drawn from that distribution,

\[ E[f] \approx \frac{1}{N} \sum_{n=1}^{N} f(x_n) \]

This approximation becomes more accurate as \(N\) grows, and it is exactly the same averaging idea used to define MSE earlier in this module, MSE is itself an approximation to an expectation.

Variance, Defined Formally

The variance of \(f(x)\) measures how much \(f(x)\) varies around its own expected value,

\[ \text{var}[f] = E\left[\left(f(x) - E[f(x)]\right)^2\right] = E[f(x)^2] - E[f(x)]^2 \]

The first form is the direct definition, average squared distance from the mean. The second form is algebraically equivalent and is often easier to compute, average of the square minus the square of the average.

This is the exact same variance used in the bias variance decomposition. \(\text{Var}(\hat{f}(x_0))\) on slide 12 is precisely this quantity, applied to the random variable \(\hat{f}(x_0)\), where the randomness comes from which training set happened to be drawn.

Covariance

Covariance measures whether two random variables tend to move together. For two variables \(x\) and \(y\),

\[ \text{cov}[x, y] = E_{x,y}[xy] - E[x]E[y] \]

If \(\text{cov}[x,y] > 0\), the two variables tend to be above their averages together and below their averages together. If \(\text{cov}[x,y] < 0\), one tends to be above average when the other is below. If \(\text{cov}[x,y] = 0\), there is no linear tendency either way.

For two vectors of random variables \(\mathbf{x}\) and \(\mathbf{y}\), the same idea extends to a covariance matrix,

\[ \text{cov}[\mathbf{x}, \mathbf{y}] = E_{\mathbf{x},\mathbf{y}}\left[\mathbf{x}\mathbf{y}^{T}\right] - E[\mathbf{x}]E[\mathbf{y}]^{T} \]
Relatable example: rainfall and river level at a monitoring station typically have positive covariance, when one is unusually high, the other tends to be as well.

Putting It Together, A Worked Walkthrough

Consider a microfinance institution building a model to predict whether a borrower will default, using income, repayment history, and loan size.

  1. Framework: \(Y = f(X) + \varepsilon\), where \(Y\) is default status and \(\varepsilon\) captures factors never recorded, such as a sudden family medical expense
  2. Accuracy measure: since \(Y\) is qualitative, the team tracks the error rate on a held out test set of borrowers, not just the training set
  3. Bias variance check: a very simple rule, such as default if loan size exceeds a fixed threshold, likely has high bias. A highly flexible model fit to a small loan book likely has high variance
  4. Benchmark: the team recognizes that some defaults are essentially unpredictable in advance, an irreducible error and a non zero Bayes error rate, so they do not chase a perfect score
  5. Probability: Bayes' theorem lets the team convert a model of how income and history look given default status into the more directly useful probability of default given income and history
This is the pattern for the rest of the course: every method in Modules 3 through 13 is judged using test error, understood through bias and variance, and grounded in the same probability tools introduced here.
